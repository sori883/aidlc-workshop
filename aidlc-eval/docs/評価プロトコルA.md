# A領域 評価プロトコル（開発効率・品質: 完了済みAI-DLC開発の事後検査）

対象リポジトリ: https://github.com/sori883/aidlc-workshop
位置づけ: AI開発目標（sp_1）の「A. 開発効率・品質」に対応。実行ハーネスは `aidlc-eval/`（00〜05番のスクリプト。同梱README参照）。

## 測定方針

**新規セッションは走らせない。** 既に完了したAI-DLC開発の成果物とセッションログを事後検査し、
各ゲート/フェーズが機能していたかをプロセス内在指標の**絶対値**で記録する
（素のClaude Codeとの比較はしない）。速度・コスト系は比較対象を持たないため良否判定をせず
**観測値**として記録する。

検査は**全自動**（`./00_run_A.sh <対象>` 1コマンド）。人間の作業は完了後の採点サンプル検証のみ。

## 実行手順（詳細は aidlc-eval/README.md）

```bash
cd aidlc-workshop/aidlc-eval
./00_run_A.sh brown-field    # 全自動（green-field も同様）
```

各工程（01解決 → 02ログ解析 → 03品質 → 04AI採点 → 05レポート）は単体でも再実行できる。
生成済み出力はスキップされるため、失敗時は再実行すれば続きから走る。

## 妥当性の担保（結果の解釈ルール）

1. **サンプル検証**: `A_採点_*.md` の判定理由の15%程度を人間が検算。同意できなければ
   採点基準（採点ヘッダ・曖昧点リスト）を直し、該当採点ファイルを消して再採点。
2. **絶対値の読み方**: 比較対象がないため、良否は「各ゲートが機能していたか」
   （要件の取りこぼし・記録の穴・曖昧点の見逃しの有無）で読む。速度・コストは観測値。
3. **D4事後検査の限界**: 曖昧点リストは事後定義のため、統制された検出率（仕込み型）とは
   意味が異なる。「実際にあった曖昧さをプロセスが拾えたか」の記録として扱う。
4. **再実行**: 検査はいつでも同一入力から再現できる（セッションログ・コミットが不変のため）。
   採点モデル/バージョンを結果と一緒に記録する。

## シールド

- A用曖昧点リストは対象ごとに `shield/<対象>/A_D4_曖昧点.md` に置く（採点専用・事後定義の正解）。
  将来の注入型D2/D4資材（旧設計のお題・注入文・仕込みリスト）は
  `aidlc-eval/prompts/future/`・`shield/future/` に保管。
- マスターは `aidlc-eval/shield/`（このハーネス内で自己完結）。
