# C・D領域 評価プロトコル

対象リポジトリ: https://github.com/sori883/aidlc-workshop
評価対象ワークスペース: シナリオごとに `scenarios/<シナリオ>/scenario.json` の `workspace` で定義する
（同梱シナリオ: `brown-field/`＝定期予約のbrownfield追加 / `green-field/`＝新規構築 /
`smoke-test/`＝ハーネス動作確認用ダミー）。

位置づけ: AI開発目標（sp_1）の「C. コンテキスト喪失耐性」「D. 説明責任」

## シナリオ機構（コアとシナリオ定義の分離）

実行・採点・集計のコア（07_setup.sh / 08_run.sh / 10_grade.sh / 11_report.py）は
**シナリオ非依存**で、以下はすべて `scenarios/<シナリオ>/` のシナリオ定義に外出しされている:

- 対象ワークスペースのパス（scenario.json `workspace`）
- 実験条件 = 何を与え・何を除去するか（`conditions`。`.git` 除去もここに定義）
- 質問セットと模範解答・採点基準への参照（`tasks.<ID>.prompt` / `.shield` / `.grade_header`）
- 実験セッションへの投げ方プロンプト（`scenarios/<シナリオ>/prompts/*_prompt.md` の冒頭指示文）
- 試行回数（`tasks.<ID>.runs.<実行名>.trials`）
- 集計・レポート定義（問数・カテゴリ・ダミー問・差分列。`tasks.<ID>.report`）

新シナリオの追加はシナリオ定義ディレクトリを1つ増やすだけでよく、コアには手を入れない
（手順はREADME「新しいシナリオの追加手順」）。Phase 2 の障害シナリオ（障害ごとに質問・正解・
採点基準・条件が変わる）や Phase 3 の条件A/B比較（条件定義自体がシナリオごとに変わる）も、
シナリオ定義の追加（必要なら `overlay` / `setup_hook` で障害注入等の固有準備）で収容する。

## 前提と制約（このリポジトリ固有）

- **OKFナレッジは未導入**。現時点の条件Bは「コード + aidlc-docs」であり、測定できるのは **aidlc-docs の価値**。OKF導入後に同じ質問セットで再実行すれば「aidlc-docs vs aidlc-docs+OKF」の差分も取れる（質問セットは再利用可能に作ってある）。
- **audit.md のタイムスタンプは全レコード同一**（2026-07-10T01:19:57Z）のため、時間系の指標（リードタイム等）には使えない。D評価では「誰が・何を・どの順で承認したか」のみを根拠として扱う。
- 運用フェーズ（Phase 2障害対応）が未実施のため、**D質問リストは前半（開発判断系）のみ**。障害対応系の後半はPhase 2完了後に実事象ベースで追加する。

## シールド原則

`scenarios/<シナリオ>/shield/` 配下のファイル（質問・模範解答・採点基準・方針リスト）は
**実験セッションに絶対に見せない**。

- 実験セッションは作業ディレクトリの条件コピー（`$WORK_ROOT/<シナリオ>/cond-*`）内でのみ
  起動する（aidlc-eval自体をcwdにしない）。
- 質問は `prompts/*_prompt.md`（質問文のみ）を投げる。模範解答・採点基準は渡さない。
- **作業ディレクトリは必ずgitリポジトリの外に置く**（既定 `~/.cache/aidlc-eval/work`）。
  コピーから `.git` を除去しても、作業ディレクトリが aidlc-workshop リポジトリの内側にあると
  実験セッションが相対パスで親の `.git` に到達できる。reflog（`.git/logs/`）等は平文のため、
  Read/Glob/Grep しか許していなくてもコミットメッセージ・履歴が読め、親履歴には aidlc-docs と
  シールドそのものが含まれる。**sp1 の green-field 実行でこの経路のシールド破りが実際に観測された**
  （C_回答_A_1.md が `git show 878f9bd` の内容に言及）。
- 残余リスク: 実験セッションの Read ツールは原理上、任意の絶対パスを読める。作業ディレクトリを
  リポジトリ外に置くことで「近傍探索で辿り着く」経路は塞がるが、完全な封じ込めではない。
  回答がコピー外のパスを引用していないかは採点サンプル検証時に確認する。

## 実験の実行

**実行は自己完結の `aidlc-eval/` ハーネスで自動化済み**（aidlc-workshopリポジトリ直下＝green-field/brown-fieldと同じ階層に配置して使う。詳細は同梱README）:

```bash
cd aidlc-workshop/aidlc-eval
./07_setup.sh  <シナリオ>  # 実験条件を scenario.json どおり $WORK_ROOT/<シナリオ>/ に作成
./08_run.sh    <シナリオ>  # 全タスク×条件×試行の回答セッションを claude -p で実行
./10_grade.sh  <シナリオ>  # 別モデル（既定: opus）でブラインド採点
./11_report.py <シナリオ>  # 集計 → results/<シナリオ>/レポート.md
```

- 回答・採点・レポートはすべて `aidlc-eval/results/<シナリオ>/` に保存される。
- 質問・模範解答はシナリオごとに `aidlc-eval/scenarios/<シナリオ>/shield/` に同梱
  （マスターはここ。自己完結）。実行・採点のコアはシナリオ非依存で、シナリオ定義
  （scenario.json + prompts/ + shield/）を追加すれば他プロジェクト・他シナリオにも適用できる。
- 条件A（コードのみ）では aidlc-docs / .aidlc-rule-details / CLAUDE.md / docs / **.git** を除去する
  （scenario.json の conditions に定義）。`.git` を消すのは、git履歴に aidlc-docs が含まれ
  `git show` 経由でシールドが破れるため（条件Bも揃えて削除し、条件差を「ドキュメントの有無」
  だけにする）。この理由から、新シナリオの conditions でも `.git` 除去を必ず引き継ぐこと。

## C. コンテキスト喪失耐性の評価

### C-1. 理解度クイズ（Q&A方式）

**質問セット**: scenarios/<シナリオ>/shield/C_理解度クイズ.md（brown-field版・green-field版とも
本番20問 = 設計判断6 / 暗黙の仕様・制約8 / 運用3 / ダミー3。構成は scenario.json の
tasks.C.report（total / categories / dummy）に定義）

**実行マトリクス**: 2条件 × 3試行 = 6セッション。1セッション内で20問を一括で投げる。

**実験セッションへの投げ方**（毎回まっさらな新規セッション）:

```
あなたはこのリポジトリを引き継いだエンジニアです。
リポジトリ内のファイルを自由に読んで、以下の質問に答えてください。
各回答には根拠としたファイルパスを添えてください。
確信が持てない場合や、リポジトリから根拠が見つからない場合は
「分からない」と答えてください（推測で埋めないでください）。

Q1. ...
Q2. ...
```

回答は `aidlc-eval/results/<シナリオ>/C_回答_{A|B}_{試行番号}.md` に保存される。

**採点**:
1. 回答ファイルから条件ラベルを剥がし、`回答1.md`〜`回答6.md` にリネーム（対応表は別ファイルに保管）。
2. 回答生成と**別のモデル**のセッションに採点させる。渡すもの: 質問 + 模範解答 + 採点基準（シールドファイル）+ 匿名化した回答。1回答=0〜2点。
3. ダミー質問は「分からない/存在しないと回答=2点、もっともらしく捏造=0点」。
4. 人間が全体の15%（各条件9問程度）をサンプル検証し、AI採点との一致率を確認。一致率80%未満なら採点基準を改訂して全体を再採点。

**集計**: 条件別の平均スコア、カテゴリ別スコア差（B−A）、ダミー質問の捏造率差。

### C-2. 実タスク評価（D3相当）の準備物

実タスク（機能追加を新規セッションに依頼し、条件A/Bで比較）を回す際に使う基準:

- **設計整合性スコア**: （アーキテクチャ方針チェックリスト。D3実施時にshield/へ追加する） の10項目に対し違反あり/なしを判定。スコア = 5 −違反数（下限1）。採点は第三者AIにチェックリストと差分を渡して実施、人間がスポットチェック。
- **誤前提ターン数**: セッションtranscriptを第三者AIに渡し「存在しないファイル・API・仕様を前提に行動したターン」をカウント。判定基準はチェックリスト末尾に記載。
- 機能追加のお題は実行直前に決定しシールドする（候補: 「祝日スキップ機能」「隔週の定期予約」「会議室ごとの営業時間制約」— いずれも既存の recurrence 純粋関数・重複判定・レイヤ規約に触れるため整合性の差が出やすい）。

## D. 説明責任の評価

**質問セット**: scenarios/<シナリオ>/shield/D_想定質問リスト.md（brown-field版・green-field版とも
前半15問＋ダミー2問。構成は scenario.json の tasks.D.report に定義）

**実行条件**:
- **記録あり**: 新規セッションに `aidlc-docs/`（+ Vision）のみを与える（`app/`・`tests/` は渡さない。「記録だけで説明責任を果たせるか」が問いのため）。
- **記録なし**（比較対照）: コードのみ（条件Aと同一）で同じ質問に答えさせる。当初はgit logの利用を想定していたが、このリポジトリのコミットには aidlc-docs が含まれており `git show` で記録が読めてしまうため、git履歴は与えない。

**実験セッションへの投げ方**:

```
あなたはこのプロジェクトの説明責任を負う立場です。
ステークホルダーからの以下の質問に、リポジトリ内の記録を根拠に答えてください。
各回答には必ず根拠となるファイルパス（可能なら該当セクション名）を引用してください。
記録に根拠が見つからない場合は「記録なし」と明言してください。
```

**採点**（Cと同じくブラインド・第三者AI + 人間サンプル検証）:
- **回答可（2点）**: 引用付きで模範解答と整合する回答
- **部分可（1点）**: 内容は概ね正しいが根拠引用がない/薄い
- **不可（0点）**: 誤答、または「記録なし」（※模範解答側で「記録なし」が正解の質問は逆転して2点）

**副産物**: 「不可」になった質問 = **記録の穴リスト**として `aidlc-eval/results/<シナリオ>/レポート.md` に自動集計される。AI-DLC運用改善（audit.mdに何を書かせるべきか）へフィードバックする。

**機械集計指標**（Q&Aと独立に実施）:
- 承認記録カバレッジ: audit.md の承認レコード（「承認」応答）を列挙し、重要判断リスト（チェックリスト参照）に対する記録率を算出。
- draft/approved比率: OKF導入後に status メタデータを集計（現状は対象なし）。
