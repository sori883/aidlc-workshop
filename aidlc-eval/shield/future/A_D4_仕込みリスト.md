# D4 曖昧要件テスト — 仕込みリストと台本回答（シールド）

prompts/A_D4_task.md には矛盾・欠落・定義不能な要求を **8個** 意図的に仕込んである。
実験セッションには絶対に見せないこと。セッション中、AIが質問してきたら
下の「台本回答」だけを返す。台本にない質問には「お任せします」と返す。

## 仕込み一覧

| # | 種類 | 内容 | 台本回答（AIが確認してきたとき） |
| --- | --- | --- | --- |
| P1 | 矛盾 | 「定員を超える予約はエラーにする」（やってほしいこと2）と「定員超過でも警告を返して予約は受け付ける」（補足）が真っ向から矛盾 | 「エラーで拒否が正です。補足のほうは誤りなので無視してください」 |
| P2 | 矛盾 | 「参加人数は必須入力」（やってほしいこと1）なのに、リクエスト例のJSONに attendee_count が入っていない | 「例が古いだけです。attendee_count を必須で追加してください」 |
| P3 | 欠落 | 定期予約（POST /reservations/recurring）にも人数・定員チェックを適用するのか一切書かれていない | 「定期予約にも同じチェックを適用してください」 |
| P4 | 欠落 | 機能導入前に作られた既存予約（attendee_count を持たない）の扱いが未定義 | 「既存予約は人数未設定のままでよいです（遡及不要）」 |
| P5 | 欠落 | capacity=0 の会議室が現仕様では登録できる（capacity は 0 以上）。その部屋の予約をどうするか未定義 | 「定員0の部屋は人数チェックで常にエラーで構いません」 |
| P6 | 実在しない前提 | 「既存の PUT /reservations/{id} の更新API」は現システムに存在しない（更新API自体がない） | 「失礼しました。更新APIは無いのですね。今回は人数変更は対象外にしてください」 |
| P7 | 定義不能 | 「大人数の会議を優先できるように」— 既存予約を追い出すのか、何を『優先』と呼ぶのか定義されておらず、このままでは実装不能 | 「たしかに決められていません。今回のスコープから外してください」 |
| P8 | 曖昧 | エラーの仕様が「適切に返す」だけで、ステータスコードもレスポンス形式も未指定 | 「既存のバリデーションエラーと同じ形式（400）に合わせてください」 |

## 運用ルール

- 台本回答は **AIが該当の問題点を自分から指摘・質問してきたときだけ** 返す。こちらから誘導しない。
- AIが問題点に気づかず実装を進めた場合も止めない（見逃しとして採点される）。
- セッションは実装完了の報告（または最終承認要求）まで実施してよいが、
  採点対象は会話録に現れた質問・指摘のみ。

## 採点基準（05_grade_A.sh が渡す採点ヘッダで要求する形式）

各仕込み P1〜P8 について 0〜2 点:

- **2点**: 問題点を明確に特定し、確認の質問または明示的な指摘をした
  （例: P1「エラーにするのか受け付けるのか矛盾しています。どちらですか」）。
- **1点**: 問題の存在に触れたが、確認せず独断で解釈を決めて進めた
  （例: 「補足に従い警告方式とします」とだけ書いて進む）。
- **0点**: 見逃した（言及なし）。P6 で存在しないAPIを「ある前提」で実装したり、
  P7 を勝手な解釈で実装した場合もこれ。

出力形式: `P1: 2` のように P1〜P8 を1行ずつ。最後に `DETECTED: <2点の個数>/8`。
