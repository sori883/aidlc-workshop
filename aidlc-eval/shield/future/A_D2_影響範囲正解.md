# D2 仕様変更「キャンセルは開始30分前まで」— 影響範囲の正解定義（シールド）

対象: brown-field（会議室予約API）。prompts/A_D2_inject.md の変更依頼に対する正解。
実験セッションには絶対に見せないこと。

## 正解の影響範囲（コア4項目）

AIの列挙がこの4項目を押さえているかで採点する。ファイルパスと「何を変えるか」の
両方が合っていて的中（HIT）。パスだけ・内容だけなら部分的中（採点者の判断で HIT 0.5 換算はせず、
迷ったら MISS に倒す）。

1. **`app/reservations/service.py` — `cancel_reservation`**
   30分前チェックの追加。順序が重要: 既に cancelled の予約は時刻に関わらず200（冪等維持）なので、
   冪等分岐をチェックより先に通す必要がある。この順序への言及があれば質の高い的中。
2. **`app/series/service.py` — シリーズキャンセル（`cancel_series` 相当）**
   キャンセル対象回のフィルタ変更: 「start_time > now の active 回」→「start_time − 30分 > now の active 回」。
   制限を過ぎた回はスキップ（エラーにしない）。
3. **`tests/test_reservations_api.py` — 単発キャンセルのテスト**
   既存のキャンセルテストは「未来の予約」を作ってキャンセルしているため、30分境界を跨ぐケースの
   追加・修正が必要（ちょうど30分前は不可、30分超前は可、cancelled への再キャンセルは常に200）。
4. **`tests/test_recurring_api.py` — シリーズキャンセルのテスト**
   直近回（30分以内に開始）を含むシリーズでスキップ動作を検証するテストの追加・既存修正。

## ボーナス（EXTRAに数えない正当な言及）

- `README.md` / `aidlc-docs` 内の「時間制約のキャンセルポリシーはスコープ外」という記述の更新
  （設計上の割り切りが変わるため）。
- 開発中の新機能との相互作用への言及
  （F1: 予約変更APIにも同様の時間制限を入れるべきか確認する、など）。
- 会議室削除（active予約があると409）への間接影響の考察
  （キャンセルできない予約が増えるため削除できない期間が延びる）。

## 影響なし（列挙されたらEXTRA=過剰に数える）

- `app/availability/`（重複判定・空き検索はキャンセルポリシーと無関係）
- `app/rooms/`（削除ガードのロジック自体は不変）
- `app/db/models.py` / スキーマ変更（新しい列は不要）
- `app/series/recurrence.py`（発生日生成は無関係）

## 採点出力（05_grade_A.sh が渡す採点ヘッダで要求する形式）

- `HIT: <n>`   … コア4項目のうち列挙できた数（0〜4）
- `MISS: <n>`  … 漏らした数（4 − HIT）
- `EXTRA: <n>` … 「影響なし」リストに該当する過剰列挙の数（ボーナス言及は数えない）
- `REGRESSIONS_TRUE: <n>` … 参考として渡される baseline テスト失敗一覧のうち、
  この仕様変更で失敗して当然のもの（上記3・4のテスト）を除いた「真の退行」数。
  失敗一覧が渡されない場合は `REGRESSIONS_TRUE: -1`。
