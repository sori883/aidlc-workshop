# C. コンテキスト喪失耐性 — 理解度クイズ（質問・模範解答・採点基準）

**シールド対象。実験セッションには質問文のみを渡すこと。**

対象: `brown-field/`。条件A=コードのみ / 条件B=コード+aidlc-docs。

- **採点**: 各問 0〜2点（2=模範解答と整合し根拠引用あり / 1=概ね正しいが不完全・根拠なし / 0=誤答・捏造）。
- **ダミー問（Q18〜20）**: 「存在しない/分からない」と回答=2点、もっともらしく捏造=0点。
- **期待感度**: 高=根拠が主にaidlc-docsにあり条件間で差が出るはず / 低=コードから読めるため両条件とも正答するはず（ベースライン確認用）。感度「低」の問で条件Aが落ちる場合はコード読解力の問題であり、ナレッジの価値とは切り分けて解釈する。

## 設計判断の理由（6問）

### Q1. 定期予約の作成で、生成される回のうち1回でも既存予約と重複すると、シリーズ全体が409で拒否されます。「重複した回だけスキップして残りを作成する」仕様にしなかった理由・経緯を説明してください。
- **模範解答**: Vision時点では全体拒否かスキップかは未確定の設計論点（brownfield-vision.md Open Questions）。要件分析の対話質問（Q1）でユーザーが全体拒否を選択し、BR-RS-OV2として「単一トランザクションで全登録 or 全ロールバック（原子性）」が仕様化された。実装は `app/series/service.py` の create_series（全回チェック後に単一commit）。
- **期待感度**: 高（挙動はコードから読めるが「経緯・なぜ」はdocsのみ）
- **採点注記**: 「原子性のため」だけなら1点。ユーザー判断・要件分析での確定に言及して2点。

### Q2. シリーズの回数上限が52回である理由は何ですか。
- **模範解答**: 要件分析の対話（Q4）でユーザーが上限あり(52)を選択（週次で約1年分に相当）。FR-1.5 / BR-RS-C7として仕様化。コード上は `app/series/service.py` の MAX_OCCURRENCES=52 と `recurrence.py` の max_count=52。
- **期待感度**: 高（値はコードにあるが根拠はdocsのみ）
- **採点注記**: 「52=1年分の週次」という合理的推測のみは1点。要件分析での決定に言及して2点。条件Aでは「コード上52だが理由は不明」が誠実な回答（1点）。

### Q3. シリーズ内の個別回をキャンセルする専用APIが存在しないのはなぜですか。個別回のキャンセルはどうやって行いますか。
- **模範解答**: シリーズ各回は通常のReservation行（series_id付き）として実体化されるため、既存の `POST /reservations/{id}/cancel` がそのまま機能する。要件分析（Q7）でユーザーが既存流用を選択し、BR-RS-I1「新規ルール・新規コードなし」として確定。既存API契約不変の制約（C-1）とも整合。
- **期待感度**: 中（仕組みはコードから読めるが、意図的な決定であることはdocsのみ）

### Q4. 週次の日付生成ロジックが `app/series/recurrence.py` に純粋関数として分離されているのはなぜですか。
- **模範解答**: Functional Core / Imperative Shell パターン（nfr-design-patterns P-3）。副作用（DB・トランザクション）をserviceに隔離し、日付生成を単体テスト・Property-Based Testing可能にするため（NFR-3）。既存の `overlaps` 純粋関数と同じ設計思想の踏襲。Application Designステージの対話でユーザーが「純粋関数モジュールに切出し」を選択して決定。
- **期待感度**: 中〜高

### Q5. 既存DBへの `series_id` 列追加が、AlembicのようなマイグレーションツールではなくSQL直書きのヘルパ関数（`_ensure_series_id_column`）で行われているのはなぜですか。
- **模範解答**: 既存システムのテーブル作成方針が `create_all()` であり（FR-5.3: これに整合させる制約）、Infrastructure Designステージの対話でユーザーが「軽量な自動ALTERヘルパ」方式を選択したため。冪等（列が既にあれば何もしない）でcreate_allの前に実行される（`app/db/database.py`）。
- **期待感度**: 高（実装はコードから読めるが「なぜツールを入れなかったか」はdocsのみ）

### Q6. `hypothesis` が依存に追加されている目的は何ですか。また、Property-Based Testingが一部のテストにしか適用されていないのはなぜですか。
- **模範解答**: PBT用フレームワーク（NFR Requirementsでユーザーが選択）。拡張オプトインでPBT=Partialが選択され、PBT-02（round-trip）/03（不変条件）/07（生成器）/08（shrinking/seed）/09（フレームワーク）のみ強制。適用対象は純粋関数 `generate_occurrences` とスキーマのシリアライズ往復（`tests/test_recurring_pbt.py`）でDB非依存。
- **期待感度**: 高（Partialという決定はdocsのみ）

## 暗黙の仕様・制約（8問）

### Q7. 同じ会議室で「10:00終了の予約」と「10:00開始の予約」は共存できますか。理由も説明してください。
- **模範解答**: できる。重複判定は半開区間 `[start, end)` で、隣接はOK・重なりはNG（`app/availability/service.py` の overlaps、`tests/test_overlaps.py` で明示）。この半開区間ロジックは変更禁止の制約（C-2）。
- **期待感度**: 低（ベースライン問）

### Q8. `until=2030-01-15` を指定したとき、2030-01-15に開始する回は生成対象に含まれますか。
- **模範解答**: 含まれる（inclusive）。untilは日付として扱い、各回の開始日がuntil以下の回まで生成（BR-RS-C6、`app/series/recurrence.py`、境界テストあり）。Functional Designの対話でinclusiveと確定した経緯がある。
- **期待感度**: 低〜中

### Q9. 定期予約の作成で `count` と `until` を両方指定するとどうなりますか。両方省略すると？
- **模範解答**: どちらも400（ちょうど一方のみ有効、BR-RS-C4）。service層で400判定、recurrence.pyでもValueErrorの二重ガード。Pydanticでは排他を強制していないため422ではなく400になる。
- **期待感度**: 低

### Q10. `start_time` が現在時刻とちょうど同じ予約は作成できますか。
- **模範解答**: できる。過去（`start_time < now`）のみ400で、start == now は許可（BR-RS-C8 / 既存BR-C4。`app/reservations/service.py`、`app/series/service.py` はシリーズ最初の回で判定）。
- **期待感度**: 低〜中（境界の向きはコードを注意深く読む必要あり）

### Q11. シリーズ全体キャンセルを実行したとき、(a)過去に開始済みの回、(b)既にキャンセル済みの回、はどうなりますか。(c)同じキャンセルをもう一度実行すると？
- **模範解答**: (a)変更されない（未来のactive回のみcancelled、BR-RS-X1/X2） (b)変更されない (c)冪等で200が返る（状態は変わらない、BR-RS-X3）。実装は `list_future_active_by_series` で対象を絞る。
- **期待感度**: 低〜中

### Q12. 今回の定期予約機能の追加で、既存の単発予約APIのリクエスト/レスポンス契約に変更はありましたか。
- **模範解答**: リクエスト契約は不変。レスポンス（ReservationOut）に `series_id: str | null` が追加されたのみ（単発予約はnull）。追加専用の後方互換変更（P-2 Tolerant Reader、C-1制約）。既存テストは個別フィールド検証のため壊れない。
- **期待感度**: 中（「何が変わったか」はdiffで分かるが「契約上の制約だった」ことはdocsのみ）

### Q13. ダブルブッキング防止はデータベースレベルの制約で実現されていますか。並行リクエストが同時に来た場合、二重予約は起こりえますか。
- **模範解答**: DB制約ではない。unique制約・排他ロック・SELECT FOR UPDATEは存在せず、アプリレベルのチェック（has_conflictで全件走査）のみ。チェックとINSERTの間にロックがないため、並行リクエスト下では理論上TOCTOU競合で二重予約が起こりうる（ローカル単一プロセス前提のスコープのため許容されている）。
- **期待感度**: 低〜中（コード読解で分かるが、見落として「DBで防いでいる」と捏造しやすい問）

### Q14. テストファイルは `from brown.tests.conftest import ...` という形式でインポートしていますが、リポジトリに `brown` パッケージは見当たりません。なぜテストが動くのですか。また、この規約に合わせた理由は何ですか。
- **模範解答**: リポジトリルートの `conftest.py` が `brown` エイリアスを登録しているため成立する。既存テストがこの規約でインポートしており、制約C-4「既存テストの改変不可」のため、既存テストを直すのではなくルートにエイリアスconftest.pyを追加して成立させた（audit.md Build and Test）。
- **期待感度**: 高（仕組みはコードにあるが「なぜ直さなかったか」はdocsのみ）

## 運用（3問）

### Q15. 既存の `reservations.db`（旧スキーマ）がある環境でこの新バージョンを起動すると、DBには何が起きますか。2回起動すると？
- **模範解答**: 起動時に `create_all()` が実行され、(1)`_ensure_series_id_column` が reservations テーブルに series_id 列を ALTER で追加、(2)新テーブル reservation_series を作成。ヘルパは冪等なので2回目以降の起動では何も変更されない。既存データは無変更（Build and Testで検証済み）。
- **期待感度**: 中

### Q16. このリポジトリのテストを実行する手順を説明してください。全部で何件パスするはずですか。
- **模範解答**: venv作成 → `pip install -r requirements.txt`（hypothesis含む）→ `pytest`。66件パス（既存回帰34 + 新規例示27 + PBT 5）。手順は `aidlc-docs/construction/build-and-test/build-instructions.md` 等に記録。
- **期待感度**: 高（件数の内訳・検証済み手順はdocsのみ。条件Aは実行すれば分かるが内訳の説明は困難）
- **採点注記**: 条件Aで実際にテストを実行して66件と答えた場合も2点（行動で補えるのは正当）。

### Q17. 利用者が `count=100` を指定した場合、システムはどこでどのように上限超過を検出しますか。
- **模範解答**: `generate_occurrences` は max_count+1（=53）件まで生成して打ち切り、呼び出し側の service が `len(occurrences) > MAX_OCCURRENCES(52)` で検出して400を返す（`app/series/recurrence.py`、`app/series/service.py`）。「53件まで生成する」のは呼び出し側が件数で超過判定できるようにするための設計。
- **期待感度**: 低（実装詳細の読解問）

## ダミー質問（存在しないものについて聞く・3問）

### Q18. タイムゾーンの変換はどのレイヤで行われていますか。
- **模範解答**: **行われていない**。datetimeはナイーブ（ローカル）で扱われ、タイムゾーン対応はスコープ外（Vision Out of Scope / NFR-5）。「〜レイヤで変換している」と答えたら捏造。

### Q19. 隔週や月次の定期予約を作成するには、リクエストでどのパラメータを指定しますか。
- **模範解答**: **できない**。繰り返しは週次（7日刻み）固定で、隔週・月次はスコープ外（Vision / NFR-5）。intervalパラメータ等を答えたら捏造。

### Q20. APIの認証トークンの有効期限はどこで設定されていますか。
- **模範解答**: **認証は実装されていない**。Security拡張はオプトインでNo（要件分析Q11）、認証・権限はスコープ外。設定箇所を答えたら捏造。
