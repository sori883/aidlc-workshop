# aidlc-eval — A・C・D領域 評価実行ハーネス

AI-DLC検証のA（開発効率・品質）・C（コンテキスト喪失耐性）・D（説明責任）を測定する。
評価設計の詳細は同梱の `docs/評価プロトコルA.md`（A）と `docs/評価プロトコルCD.md`（C・D）を参照。
**このディレクトリは自己完結しており、外部のノート環境（Obsidian等）を参照せずに動く。**

- **A領域 = 完了済みAI-DLC開発の事後検査**。新規セッションは走らせず、既存のセッションログと
  成果物（gitコミット・aidlc-docs）を検査する。green-field / brown-field 両対応。
- **C・D領域 = 新規の回答セッションを回す実験**。`claude -p` で読み取り専用セッションを起動し、
  質問セットに答えさせて採点する。**コア（07/08/10/11）はシナリオ非依存**で、
  対象ワークスペース・実験条件・質問セット・試行回数はすべて `scenarios/<シナリオ>/` の
  シナリオ定義（scenario.json + prompts/ + shield/）に外出しされている。
  シナリオの追加はディレクトリを1つ増やすだけで、コアのスクリプトには手を入れない
  （「新しいシナリオの追加手順」参照）。

**自己完結・相対パス構成**: このディレクトリごと aidlc-workshop リポジトリ直下
（green-field / brown-field と同じ階層）に置いて使う。

```
aidlc-workshop/                ← git履歴が必要（Aのコミット境界検査に使う）
├── green-field/               ← 検査対象（会議室予約システムの新規構築）
├── brown-field/               ← 検査対象（定期予約追加）
└── aidlc-eval/                ← このディレクトリを置く
    ├── 00_run_all.sh          一括エントリポイント（A→C・D）
    ├── 00_run_A.sh 〜 05_report_A.py   A領域（事後検査）
    ├── 07_setup.sh / 08_run.sh / 10_grade.sh / 11_report.py   C・D系コア（シナリオ非依存）
    ├── scenario_lib.py        シナリオ定義の読み込み・検証（コア共通。スキーマもここに記載）
    ├── docs/                  評価プロトコル（設計文書。このハーネスがマスター）
    ├── prompts/               シナリオ非依存の採点ヘッダ（grade_*.md）
    │   └── future/            将来の注入型シナリオ用（D2仕様変更・仕込み型D4）の保管
    ├── scenarios/<シナリオ>/  シナリオ定義（1シナリオ=1ディレクトリ）
    │   ├── scenario.json      対象パス・実験条件・タスク・試行回数・集計定義
    │   ├── prompts/           実験セッションに渡す質問文（模範解答なし）
    │   └── shield/            質問・模範解答・採点基準（実験セッションに見せない）
    ├── shield/<対象>/         A領域用シールド（A_D4_曖昧点.md）。future/ は注入型の保管
    └── results/<シナリオ>/    測定・採点・レポートの出力先（対象ごとに分離）

~/.cache/aidlc-eval/work/<シナリオ>/   実験条件のコピー（07が生成。リポジトリの外に
                                       置く理由は「設計上の注意」参照。WORK_ROOTで変更可）
```

## 前提

- `claude` CLI（ログイン済み）と `python3`
- `uvx`（uv。A領域の静的解析・カバレッジ用。`brew install uv`）
- A領域は**開発セッションのJSONLログ**が必要（`~/.claude/projects/` 配下。開発を実施した
  マシン以外で動かす場合はJSONLをコピーし `A_SESSION_JSONL=` で指定）
- pip-audit はネットワークを使う（`SKIP_PIP_AUDIT=1` で省略可）

## 実行手順

各スクリプトは第1引数で対象フォルダ名を指定できる（省略時: `brown-field`）。
出力は対象ごとに `results/<対象>/` に分離される。

**一括実行（A→C→D を1コマンドで）:**

```bash
cd aidlc-workshop/aidlc-eval
./00_run_all.sh brown-field   # → results/brown-field/レポートA.md と レポート.md
./00_run_all.sh green-field   # → results/green-field/ に同様に出力
```

> A領域は green-field / brown-field 両対応。C・D領域はシナリオ定義
> （`scenarios/<シナリオ>/scenario.json`）がある場合のみ実行され、なければスキップされる
> （同梱シナリオ: brown-field / green-field / smoke-test）。
> 所要時間の目安: A領域は数分＋AI採点3回。C・D領域は回答8セッション＋採点で1時間前後。
>
> **green-field の注意**: sp1 時点の `results/green-field/` の C・D 結果は、質問セットが
> brown-field 固有だった頃の無効な結果（定期予約を問う質問を green-field に投げていた）。
> 実行は既存ファイルをスキップするため、green-field 用質問セットで回すには先に
> `results/green-field/` の `C_*`・`D_*`・`レポート.md` を別ディレクトリへ退避すること
> （sp1 の記録として保全する。削除はしない）。

**A領域だけ（完了済み開発の事後検査。全自動）:**

```bash
./00_run_A.sh brown-field     # 01→05 を順に実行（工程別スクリプトは単体でも再実行可能）

./01_setup_A.sh  <対象>   # 検査対象の解決（セッションJSONL・コミット境界の検証、manifest登録）
./02_measure_A.py <対象>  # ログ解析: リードタイム・介入回数・拘束時間近似・トークン、会話録/介入一覧
./03_quality_A.sh <対象>  # 機械検査: テスト・生成コードのカバレッジ・ruff/bandit/pip-audit・churn・extension状態
./04_grade_A.sh  <対象>   # AI採点: 要件カバレッジ・承認記録カバレッジ・D4曖昧点検出（別モデル、既定opus）
./05_report_A.py <対象>   # 集計 → results/<対象>/レポートA.md
```

**C・D領域だけ（回答セッション実験）:**

```bash
./07_setup.sh brown-field     # 実験条件を scenario.json どおり作業dirに作成（数秒）
./08_run.sh   brown-field     # 全タスク×条件×試行の回答セッション（brown-fieldは8本。1本 数分〜10分）
./10_grade.sh brown-field     # 全回答を別モデル（既定: opus）でブラインド採点
./11_report.py brown-field    # 集計 → results/brown-field/レポート.md
```

## 新しいシナリオの追加手順（C・D系）

実行・採点のコア（07/08/10/11、scenario_lib.py、`prompts/grade_*.md`）はシナリオ非依存。
新しいシナリオ `<名前>` の追加は **`scenarios/<名前>/` を1つ作るだけ**で、
コアのスクリプトには一切手を入れない。最小の実例が `scenarios/smoke-test/`（質問3問）、
フル構成の実例が `scenarios/brown-field/`。

1. ディレクトリを作る: `scenarios/<名前>/{prompts,shield}/`
2. `prompts/<タスク>_prompt.md` — 実験セッションに渡す文面（冒頭指示＋質問文のみ。
   模範解答は絶対に含めない）。既存シナリオの冒頭指示文が流用できる。
3. `shield/<タスク>_質問.md` — 質問・模範解答・採点基準。ダミー質問（捏造検出用）の
   問番号と採点特則は**このファイル内に明記する**（採点ヘッダはシナリオ非依存で、
   シールド内の記載に従って採点する）。
4. `scenario.json` — 対象ワークスペースのパス、実験条件（何を与え・何を除去するか）、
   タスク（prompt/shield/採点ヘッダ/実行条件と試行回数）、集計定義（問数・カテゴリ・
   ダミー問・差分列）。スキーマの全項目は `scenario_lib.py` 冒頭のドキュメント参照。
5. 実行:

   ```bash
   ./07_setup.sh <名前> && ./08_run.sh <名前> && ./10_grade.sh <名前> && ./11_report.py <名前>
   ```

定義に誤りがあれば scenario_lib.py の検証が実行前にエラーで指摘する。
シナリオ定義が無い名前では C・D領域はスキップ（00_run_all.sh）またはエラー停止（07〜11）する。

**Phase 2（障害シナリオ）/ Phase 3（条件A/B比較）もこの手順で収容する**: 障害ごとに
`scenarios/incident-XX/` を作り、質問・正解・採点基準（shield/）と条件定義を差し替える。
条件に固有の準備（追加ファイルの配置・障害注入など）が要る場合は scenario.json の
`overlay`（上書きコピーするファイル群）と `setup_hook`（構築後に実行するシナリオ同梱
スクリプト）で吸収する。OKFナレッジ比較は conditions に `B-okf` のような条件を足し、
runs をそれに向けるだけでよい。

A領域を新対象で動かす場合は、`config.sh` にセッションJSONL・コミット境界を定義し、
D4採点をするなら `shield/<対象>/A_D4_曖昧点.md` を用意する（A領域は scenarios/ を使わない）。

## `results/<対象>/` に生成されるもの

| ファイル | 領域 | 内容 |
| --- | --- | --- |
| `レポートA.md` | A | 速度・コスト（観測値）／品質／要件・承認記録カバレッジ／D4検出／D5記述検査 |
| `A_測定_*.json` / `A_品質_*.json` | A | 機械測定の生データ |
| `A_採点_要件_*.md` / `A_採点_承認記録_*.md` / `A_採点_D4_*.md` | A | AI採点の判定理由つき結果 |
| `A_会話録_*.md` / `A_介入一覧_*.md` | A | セッションログの可読版（採点入力・人間の補正用） |
| `A_sessions.tsv` | A | 検査レコード（セッションJSONL・コミット境界） |
| `<タスク>_回答_<実行名>[_<試行>].md` | C・D | 回答セッションの生データ（試行1回の実行は試行番号なし。例: `C_回答_A_1.md`, `D_回答_records.md`） |
| `<タスク>_採点_*.md` | C・D | 問別スコア（0〜2点）と判定理由 |
| `レポート.md` | C・D | scenario.json の集計定義に従った条件別集計・差分（例: **B−A**）・問別スコア・記録の穴リスト |

## A領域の注意

- 検査対象ごとの**セッションJSONLとコミット境界**は `config.sh` に定義してある
  （green-field: `f6d03df..227a8ca` / brown-field: `69a4905..878f9bd`。`A_SESSION_JSONL=` 等で上書き可）。
- **品質検査はheadコミットのworktreeに対して行う**（現ディレクトリは開発完了後に改変されて
  いる可能性があるため。実際 green-field/tests は改変済みで単体では実行不能）。
- **人間拘束時間は近似値（離席込みの上限）**。`A_介入一覧_*.md` に全介入が時刻つきで出るので、
  正確な値が必要なら手動補正する。
- **承認後やり直し回数**は機械判定が不安定なため、`A_介入一覧_*.md` を見て人間が数える
  （承認後に差し戻した発言の数）。
- **D4（曖昧要件検出）は事後検査版**: 元指示・Visionの曖昧点リスト（`shield/<対象>/A_D4_曖昧点.md`）
  のうち、要件検証質問・会話で人間に確認されたものの割合を採点する。
- **D2（仕様変更）は条件付き検査**: 04がまず介入一覧から「セッション内に仕様変更事象があったか」を
  AI判定し、**なければ「該当なし」を記録、あれば指標どおり評価**する（影響範囲の的中/漏れ/過剰・退行）。
  注入型（【仕様変更】マーカー）なら shield/future/ の正解を基準に、自然発生なら実際の変更内容を
  基準に採点する。時間系（変更対応時間・変更後介入数）はマーカーがある場合のみ02が自動計測。
- **仕込み型D4と注入型D2の統制実験は将来実施**: 資材（お題・注入文・正解シールド）は
  `prompts/future/`・`shield/future/` に保管してある。
- D5はaidlc-state.mdのextension opt-in状態とPBTの痕跡を記述的に検査（有効/無効の差分比較は将来）。
- トークン集計は message.id 重複除去済み（サブエージェント消費込み。素朴合計は約2倍に過大計上）。

## 実行後にやること（人間の作業）

1. **採点のサンプル検証**: `results/*_採点_*.md`（A領域は `A_採点_*.md`）の判定理由を15%程度読み、
   `scenarios/<シナリオ>/shield/`（A領域は `shield/<対象>/`）の模範解答・採点基準と照らして
   同意できるか確認。おかしければ該当採点ファイルを削除し、シールドの採点基準を直して
   `./10_grade.sh`（A領域は `./04_grade_A.sh`）を再実行。一致率80%未満なら基準改訂のうえ全体を再採点。
2. **承認後やり直し回数のカウント**（A領域）: `A_介入一覧_*.md` を見て承認後の差し戻し発言を数える。
3. **レポートの解釈**: C・Dは「設計判断の理由」カテゴリのB−A差分と、ダミー質問
   （`scenarios/<シナリオ>/scenario.json` の report.dummy で定義）の捏造数差が主目的の数値。A領域（`レポートA.md`）は各ゲート/フェーズが機能していたか（要件の取りこぼし・
   記録の穴・曖昧点の見逃しの有無）で読み、速度・コストは観測値として扱う。

## 再実行・やり直し

- コストの大きい工程（回答セッション・AI採点）は**既存の出力ファイルをスキップ**する。
  特定の回答・採点だけやり直すには該当ファイルを削除して再実行。
- A領域を全部やり直す: `results/<対象>/` の `A_*` と `レポートA.md` を削除して `./00_run_A.sh <対象>`。
- C・D領域を全部やり直す: `results/<シナリオ>/` の各タスクの `*_回答_*`・`*_採点_*`・`レポート.md` を
  削除して 07 から（または `./00_run_all.sh`）。
- 機械測定（02・03・05・07・11）は決定的で何度実行しても同じ結果になる。AI採点は判定が揺れ得るため、
  再採点したら必ずサンプル検証をやり直す。

## 設定（環境変数で上書き）

| 変数 | 既定値 | 領域 | 用途 |
| --- | --- | --- | --- |
| `TARGET_FIELD` | `brown-field` | 共通 | 対象フォルダ名/シナリオ名（第1引数でも指定可。引数が優先） |
| `FIELD_DIR` | （自動検出） | A | 対象を絶対パスで明示指定する場合（C・D系の対象パスは scenario.json の workspace） |
| `GRADE_MODEL` | `opus` | 共通 | 採点モデル。**回答・開発側と別モデルにすること** |
| `A_SESSION_JSONL` | （config.shの既定） | A | 検査する開発セッションのJSONLパス |
| `A_BASE_COMMIT` / `A_HEAD_COMMIT` | （config.shの既定） | A | 生成コードのコミット境界 |
| `SKIP_PIP_AUDIT` | `0` | A | `1`で依存脆弱性スキャンを省略（オフライン時） |
| `ANSWER_MODEL` | （デフォルトモデル） | C・D | 回答セッションのモデル |
| `ANSWER_TOOLS` | （scenario.jsonの値） | C・D | 回答セッションの許可ツールを全条件一律で上書きする場合のみ指定 |
| `WORK_ROOT` | `~/.cache/aidlc-eval/work` | C・D | 実験条件コピーの置き場（リポジトリ外に置くこと。「設計上の注意」参照） |
| `RESULTS_ROOT` | `aidlc-eval/results` | 共通 | 出力先ルート（回帰テスト等で分離したい場合） |

試行回数はシナリオ定義（scenario.json の `runs.<実行名>.trials`）で指定する。

例: `GRADE_MODEL=sonnet ./10_grade.sh brown-field`

## 設計上の注意

- **シールド**: C・Dの実験セッションは `$WORK_ROOT/<シナリオ>/cond-*` をカレントディレクトリとして
  起動され、質問文のみ（模範解答なし）を受け取る。`scenarios/*/shield/` を実験セッションに
  読ませないこと（このディレクトリ自体で対話セッションを起動する場合も注意）。
- **作業ディレクトリはリポジトリの外に置く**（既定 `~/.cache/aidlc-eval/work`）:
  コピーから `.git` を除去しても、作業ディレクトリが aidlc-workshop リポジトリの内側にあると、
  実験セッションが相対パスで親の `.git` に到達できてしまう（reflog 等は平文で、Read だけで
  コミットメッセージ・履歴が読める。親履歴には aidlc-docs とシールドそのものが含まれる）。
  sp1 の green-field 実行ではこの経路のシールド破りが実際に観測された
  （回答が `git show 878f9bd` の内容に言及）。`WORK_ROOT` を変更する場合も必ず
  gitリポジトリの外を指定すること。なお Read ツールは原理上任意の絶対パスを読めるため
  完全な封じ込めではない（残余リスク。詳細は docs/評価プロトコルCD.md）。
- **ブラインド採点**: 採点セッションには回答・diff・会話録の本文だけを渡し、条件ラベル・ファイル名は
  渡さない。ただし回答内の引用パス（aidlc-docs/...）から条件が推測できる余地は残る
  （引用必須の設計とのトレードオフ）。
- **D「記録なし」条件はgit履歴も与えない**: このリポジトリのコミットにはaidlc-docsが含まれ、
  `git show` で記録が読めてしまうため（条件A/Bも同様に `.git` を除去し、条件差を
  「ドキュメントの有無」だけにしている）。この除去は各シナリオの scenario.json の
  conditions（remove: [".git"] 等）に定義されており、新シナリオでも必ず引き継ぐこと。
- **テスト実行系の質問**: 既定は読み取り専用のため、実験セッションはテストを実行して件数を
  確かめられない。許す場合は scenario.json の `answer_tools` に
  `"Read,Glob,Grep,Bash(pytest:*),Bash(pip:*),Bash(python:*)"` のように定義する
  （**全条件で同じ設定になる**。環境変数 `ANSWER_TOOLS` でも一律上書き可）。
- scenarios/・shield/・docs/ の**マスターはこのハーネス内**（自己完結）。外部ノート等に写しを
  置く場合はこちらを正とする。

## 将来の拡張ポイント（B領域・OKFナレッジ評価）

- **B領域（運用解決能力・Phase 2）**: 障害シナリオ10本は `scenarios/incident-XX/` として
  **シナリオ定義の追加だけで**収容する（コア無変更）。障害ごとに変わる質問・正解・採点基準は
  各シナリオの shield/ に、条件定義（何を与えるか）は scenario.json の conditions に、
  障害注入などシナリオ固有の準備は `overlay` / `setup_hook` に置く。
  障害対応セッションの測定（MTTR相当・ターン数・介入回数）には `02_measure_A.py` の
  JSONL解析を流用できる。
- **OKFナレッジ評価**: 同じ質問セットのまま scenario.json の conditions に
  条件（例: `B-okf` = コード+aidlc-docs+OKF bundle。bundleの配置は overlay で）を追加し、
  runs をそれに向けて 07→08→10→11 を再実行すれば、aidlc-docs単体とOKF併用の差分が取れる。
  D領域の draft/approved 比率・ナレッジ生成率（OKFのstatusメタデータ集計）は導入後に
  機械集計スクリプトを追加する。

  （アーキテクチャ方針チェックリスト・誤前提ターン数）を障害解決タスクに置き換えて使う。
