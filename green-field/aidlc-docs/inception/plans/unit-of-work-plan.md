# ユニット分解計画 (Unit of Work Plan)

## 目的
システムを開発単位（ユニット）に分解する。会議室予約システムは小規模・ローカル完結・モノリス制約（NFR-01）のため、単一ユニットが有力。ワークショップとして分解工程を体験する。

## 分解方針の質問（ユーザー回答）

### Q-U1: ユニット分解の方針
システムをいくつの開発ユニットに分解しますか？

A) 単一ユニット（モノリス）— 1つの FastAPI アプリ内に Room / Reservation / Availability を論理モジュールとして配置（ローカル完結・2〜3時間の制約に最適・推奨）

B) 2ユニットに分割 — 「会議室サービス」と「予約サービス」を別デプロイ単位に（マイクロサービスの体験になるが、ワークショップ範囲を超えやすい）

X) Other

[Answer]: A
