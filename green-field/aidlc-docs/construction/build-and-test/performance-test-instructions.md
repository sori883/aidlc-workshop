# Performance Test Instructions

## 適用可否
本プロジェクトは**ローカル・少人数・低同時実行**を前提とするワークショップ用途（NFR-P2 / Q-N1=A）。
明示的な性能SLOは設定しておらず、本格的な負荷試験は**スコープ外（任意）**とする。

## 参考: 簡易スモーク（任意）
負荷傾向を軽く見たい場合の一例（インストール任意）:

```bash
# 例: hey などのツールでヘルスエンドポイントを軽く叩く
# go install github.com/rakyll/hey@latest
uvicorn app.main:app --port 8000 &
hey -n 500 -c 10 http://127.0.0.1:8000/health
```

## 目安（保証値ではない）
- ローカルSQLite・単一プロセスで、CRUD/予約系の単一リクエストは体感即時（数ms〜十数ms程度）。
- 重複チェックは対象会議室の active 予約のみを走査し、`reservations(room_id, status)` インデックスで絞り込む。

## スケールが必要になった場合の方向性（今回は未実施）
- 予約数が非常に多い場合、期間フィルタや重複チェックをSQL側の範囲条件に寄せる最適化。
- 複数プロセス/本番運用が必要なら、RDBMS（PostgreSQL 等）への移行とDBレベルの排他制御を検討。

**結論**: 今回の要件では性能テストは N/A（未実施）。機能テストと統合フローの検証で品質を担保する。
