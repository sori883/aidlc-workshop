# 要件定義書 — 社内会議室予約システム

## 1. インテント分析サマリ (Intent Analysis Summary)

- **ユーザーリクエスト**: 社内向けの会議室予約システムを構築したい。現在は共有スプレッドシートで管理しており、ダブルブッキングが頻発している。
- **リクエスト種別**: 新規プロジェクト (Greenfield / New Project)
- **スコープ見積り**: 複数コンポーネント（会議室管理・予約管理・重複防止・空き検索）
- **複雑度見積り**: 中程度 (Moderate) — コア要件は明確、重複防止ロジックが要注意点
- **要件分析の深さ**: Standard

## 2. 背景・課題 (Background)

- 共有スプレッドシートによる会議室管理では、同一会議室・同一時間帯への**ダブルブッキング（二重予約）が頻発**している。
- システム化により、**予約の重複を機械的に防止**し、登録・予約・キャンセルを一貫した手段で行えるようにする。

## 3. ビジネスゴール (Business Goals)

- **G1**: ダブルブッキングをゼロにする（重複予約をシステムが拒否する）。
- **G2**: 会議室の登録・予約・キャンセルをAPI経由で確実に行える。
- **G3**: ワークショップの制約内（2〜3時間）で動作する最小限かつ実用的なシステムを完成させる。

## 4. スコープ (Scope)

### 4.1 対象 (In Scope)
- 会議室のCRUD（登録・一覧・取得・更新・削除）
- 予約の作成・一覧・取得・キャンセル
- 予約重複の検出と拒否
- 指定日時に空いている会議室の検索

### 4.2 対象外 (Out of Scope)
- 定期予約（繰り返し予約） — Q2=A により今回は実装しない
- 認証・認可・ユーザー権限管理 — Q3=A により実装しない
- キャンセルポリシー（時間制約・本人限定） — Q4=A により制約なし
- 営業時間・予約可能時間帯の制約 — Q7=A により制約なし
- フロントエンド / UI（REST APIのみ）
- 通知・カレンダー連携・承認フロー等

## 5. 機能要件 (Functional Requirements)

| ID | 要件 | 根拠 |
|---|---|---|
| FR-01 | 会議室を登録できる。会議室は **名前・収容人数・設備・場所** の属性を持つ | Q5=C |
| FR-02 | 登録済み会議室の一覧・個別取得ができる | 必須 |
| FR-03 | 会議室情報を更新・削除できる | CRUD一貫性 |
| FR-04 | 会議室・開始時刻・終了時刻・予約者情報を指定して予約を作成できる。時刻は **分単位で自由指定**（例 10:00–10:45） | Q1=A |
| FR-05 | 予約者は **名前/メール等の文字列** で識別する（認証なし） | Q3=A |
| FR-06 | **同一会議室で時間帯が重複する予約は拒否**する（ダブルブッキング防止） | G1, 必須 |
| FR-07 | 予約の一覧・個別取得ができる（会議室別・期間での絞り込みを含む） | 必須 |
| FR-08 | **予約IDを指定すれば、誰でもいつでも予約をキャンセル**できる | Q4=A |
| FR-09 | **日時（開始・終了）を指定して、その時間帯に空いている会議室を検索**できる | Q6=B |

### 5.1 重複判定ルール (FR-06 詳細)
- 2つの予約 `[start_a, end_a)` と `[start_b, end_b)` が **同一会議室**かつ時間帯が重なる（`start_a < end_b` かつ `start_b < end_a`）場合、重複とみなす。
- 境界の一致（前の予約の終了時刻＝次の予約の開始時刻、例 10:00–11:00 と 11:00–12:00）は**重複ではない**（半開区間で扱う）。
- 予約作成時に既存予約と照合し、重複する場合はエラー（HTTP 409 Conflict 等）を返して作成を拒否する。

### 5.2 入力バリデーション（基本）
- 開始時刻 < 終了時刻 であること。
- 参照する会議室が存在すること。
- 予約者名（必須文字列）が空でないこと。

## 6. 非機能要件 (Non-Functional Requirements)

| ID | 要件 |
|---|---|
| NFR-01 | **技術スタック**: Python + FastAPI + SQLite。ローカル実行で完結（外部サービス依存なし） |
| NFR-02 | **インターフェース**: REST API のみ（フロントエンド不要）。OpenAPI/Swagger は FastAPI 標準機能で自動提供 |
| NFR-03 | **開発規模**: ワークショップ向けに 2〜3時間で完成できる範囲に収める |
| NFR-04 | **データ永続化**: SQLite にデータを永続化する |
| NFR-05 | **同時実行時の整合性**: 重複予約が競合下でも作られないよう、DB制約またはトランザクションで担保することが望ましい（ワークショップ範囲では簡易実装可） |
| NFR-06 | **テスト**: 重複防止ロジックを中心に自動テストを用意する |
| NFR-07 | **セキュリティ / レジリエンシー / PBT 拡張**: いずれも無効（ワークショップ・PoC のため） |

## 7. 前提・制約 (Assumptions & Constraints)

- 単一プロセス・単一ユーザー〜少人数での利用を想定（本番運用・スケールは対象外）。
- 認証がないため、予約者情報は自己申告の文字列として扱う。
- タイムゾーンはローカル固定（単純化のため。必要なら ISO 8601 文字列で受け渡し）。

## 8. 主要要件サマリ (Key Requirements Summary)

- **コア価値**: 会議室予約時に**時間帯の重複を検出して拒否**することでダブルブッキングを防ぐ（FR-06）。
- **CRUD**: 会議室（名前・収容人数・設備・場所）と予約（会議室・時間・予約者）の管理。
- **検索**: 指定日時で空いている会議室を探せる（FR-09）。
- **割り切り**: 認証・権限・定期予約・キャンセルポリシー・営業時間制約は今回スコープ外。
- **技術**: Python + FastAPI + SQLite、ローカル完結、REST APIのみ、拡張機能なし。
