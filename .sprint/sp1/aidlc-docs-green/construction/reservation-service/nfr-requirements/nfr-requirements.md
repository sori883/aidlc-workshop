# 非機能要件 (NFR Requirements) — reservation-service

## 前提
社内・少人数・ローカル完結のワークショップ用途。拡張機能（Security/PBT/Resiliency）は無効。
以下は「軽量に洗い出した」NFR であり、過剰なエンジニアリングは行わない。

## NFR 一覧
| ID | カテゴリ | 要件 | 目標/方針 |
|---|---|---|---|
| NFR-P1 | 性能 | 単一操作の応答時間 | ローカルSQLiteで実用的に即応（体感即時）。明示的なSLOは設けない |
| NFR-P2 | スケール | 同時実行 | 少人数・低同時実行を想定（Q-N1=A）。単一 uvicorn プロセスで十分 |
| NFR-A1 | 可用性 | 稼働 | ローカル起動時のみ稼働。HA/フェイルオーバー要件なし |
| NFR-R1 | 信頼性 | データ整合性 | 予約重複を作らないこと（中核）。作成は単一トランザクションで「チェック→挿入」 |
| NFR-R2 | 信頼性 | エラーハンドリング | バリデーション/参照エラー/競合を適切な HTTP ステータスで返す（400/404/409） |
| NFR-S1 | セキュリティ | 認証・認可 | なし（要件どおり。Security拡張は無効）。予約者は自己申告文字列 |
| NFR-S2 | セキュリティ | 入力対策 | ORM(SQLAlchemy) 使用によりSQLインジェクションを回避。基本的な入力バリデーションを実施 |
| NFR-M1 | 保守性 | 構造 | レイヤ分離（router/service/repository）で責務を明確化 |
| NFR-M2 | 保守性 | テスト | pytest でユニット＋APIテスト。重複防止の境界条件を重点的にカバー（Q-N2=A） |
| NFR-U1 | ユーザビリティ | API探索性 | FastAPI 標準の Swagger UI (/docs) で全エンドポイントを試行可能 |
| NFR-O1 | 運用 | 起動 | `uvicorn` で単一コマンド起動。README に手順を記載 |

## 同時実行に関する注記（NFR-P2 / NFR-R1）
- 低同時実行前提のため、アプリ層のトランザクション内チェックで重複防止は十分実用的。
- SQLite はデフォルトで書き込みが直列化されるため、少人数環境では競合は稀。
- より強固な保証（DBレベルの排他）は今回のスコープ外（Q-F3=A で確認済み）。

## 明示的に「対象外」とするNFR
- 水平スケール / ロードバランシング / マルチインスタンス
- 認証・認可・監査ログ（アプリ機能としての）
- 災害復旧 (DR) / バックアップ自動化
- 監視・アラート基盤（クラウド）
