# Personas — 定期予約機能

## P-1: 一般予約者（社員） — 田中
- **役割**: 会議室を予約して会議を運営する社員。
- **特徴**: API または Swagger UI 経由でシステムを利用。定例会議（毎週の朝会・週次定例など）を主催する。
- **動機**: 毎回手動で予約する手間を省き、定例会議の会議室を一括で確保したい。都合が悪い回だけ個別に取り消したい。
- **関わる機能**: 定期予約の作成、シリーズ全体キャンセル、個別回キャンセル、予約照会。
- **技術レベル**: 中（REST API の基本を理解）。

## P-2: 会議室運用担当 — 佐藤
- **役割**: 会議室リソースの管理・運用を担う担当者。
- **特徴**: 予約状況を俯瞰し、ダブルブッキングが起きないことを重視する。会議室の追加・更新・削除も行う。
- **動機**: 定期予約が既存の単発予約と衝突せず、予約データの一貫性が保たれることを保証したい。シリーズがどの予約に属するか把握したい。
- **関わる機能**: 予約一覧・詳細でのシリーズ情報表示、重複防止（全体拒否）の一貫性。
- **技術レベル**: 中〜高。

## P-3: API 連携開発者 — 鈴木
- **役割**: 本予約 API を別システム（社内ポータル等）から呼び出す開発者。
- **特徴**: 既存の単発予約 API を利用したクライアントを既に実装済み。後方互換性を強く要求する。
- **動機**: 定期予約機能の追加によって既存の単発予約 API の契約が変わらないこと、レスポンスの追加フィールドが既存クライアントを壊さないことを確認したい。
- **関わる機能**: 既存 API の不変性維持、`ReservationOut` への `series_id` 追加（後方互換）。
- **技術レベル**: 高。
