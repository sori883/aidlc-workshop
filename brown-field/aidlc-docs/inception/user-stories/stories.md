# User Stories — 定期予約機能

機能ベースで編成。各ストーリーは INVEST 準拠、受け入れ基準は Given-When-Then 形式。
トレーサビリティは要件 `requirements.md`（FR/NFR/C）を参照。

---

## Epic 1: 定期予約シリーズの作成

### US-R01: 定期予約シリーズの作成
**As a** 一般予約者（P-1）
**I want** 毎週同じ曜日・時間に繰り返す会議室予約を一括で作成したい
**So that** 定例会議の会議室を毎回手動予約せずに確保できる

**トレース**: FR-1.1〜FR-1.4, FR-1.10, FR-5

**受け入れ基準:**
- **AC-1 (回数指定)**: Given 有効な会議室と起点の開始/終了時刻、When `count=4` を指定して `POST /reservations/recurring` を呼ぶ、Then 起点を1回目として7日刻みの4回分が作成され、201 で `series_id` と4件の予約（`ReservationOut` のリスト）が返る。
- **AC-2 (終了日指定)**: Given 起点 2026-08-03(月) 10:00-11:00、When `until=2026-08-24` を指定、Then 開始日が until 以下の回（8/3, 8/10, 8/17, 8/24 の4回）が作成される。
- **AC-3 (週次の一貫性)**: Given 作成されたシリーズ、When 各回を確認、Then すべて同じ会議室・同じ時刻（時・分）で、隣接回の開始日が正確に7日間隔である。
- **AC-4 (シリーズ紐付け)**: Given 作成されたシリーズ、When 各回の予約を取得、Then すべて同一の `series_id` を持つ。

### US-R02: 重複時のシリーズ全体拒否（原子性）
**As a** 一般予約者（P-1） / 会議室運用担当（P-2）
**I want** シリーズ内の1回でも既存予約と重複する場合はシリーズ全体を拒否してほしい
**So that** 部分的に歯抜けのシリーズが登録されず、ダブルブッキングも起きない

**トレース**: FR-1.6, FR-1.7, NFR-2, C-2

**受け入れ基準:**
- **AC-1 (1回重複で全体拒否)**: Given ある会議室の 8/10 10:00-11:00 に既存 active 予約がある、When 8/3 から週次4回（8/3, 8/10, 8/17, 8/24）のシリーズを作成、Then 409 が返り、**1件も**作成されない（8/3, 8/17, 8/24 も登録されない）。
- **AC-2 (原子性/ロールバック)**: Given AC-1 の 409 発生後、When 予約一覧を確認、Then 当該シリーズの予約は0件で、既存予約は変化していない。
- **AC-3 (半開区間の踏襲)**: Given 既存予約 10:00-11:00、When シリーズのある回が 11:00-12:00（隣接）、Then その回は重複と見なされず、他に重複がなければシリーズは作成される（201）。
- **AC-4 (他シリーズとの重複)**: Given 別シリーズの active 回が重なる、When 新シリーズを作成、Then 同様に全体拒否（409）。

### US-R03: 作成時の入力検証
**As a** 一般予約者（P-1）
**I want** 不正な入力は明確なエラーで拒否してほしい
**So that** 既存の単発予約と一貫したルールで安全に予約できる

**トレース**: FR-1.4, FR-1.5, FR-1.8, FR-1.9, C-3

**受け入れ基準:**
- **AC-1 (終了条件の一方指定)**: Given `count` と `until` の両方を指定、When 作成、Then 400。両方とも未指定でも 400。
- **AC-2 (回数上限)**: Given `count=53`（上限52超）、または `until` が52回を超える範囲、When 作成、Then 400。
- **AC-3 (過去日時)**: Given シリーズ最初の回の開始時刻が現在より過去、When 作成、Then 400（既存の単発予約ルールと一貫）。
- **AC-4 (時刻順序)**: Given 起点の start >= end、When 作成、Then 400。
- **AC-5 (予約者名必須)**: Given `booker_name` が空、When 作成、Then 400。
- **AC-6 (会議室存在)**: Given 存在しない `room_id`、When 作成、Then 404。
- **AC-7 (Pydantic 制約)**: Given `count` が0以下など型/制約違反、When 作成、Then 422。

## Epic 2: シリーズのキャンセル

### US-R04: シリーズ全体のキャンセル
**As a** 一般予約者（P-1）
**I want** 定例会議が終了・中止になったらシリーズの残り回をまとめてキャンセルしたい
**So that** 今後不要になった会議室予約を一度に解放できる

**トレース**: FR-2.1〜FR-2.5

**受け入れ基準:**
- **AC-1 (未来 active 回のみ)**: Given 過去回・未来回を含むシリーズ、When `POST /reservations/recurring/{series_id}/cancel`、Then 未来の active 回のみ cancelled になり、200 が返る。
- **AC-2 (過去回は不変)**: Given 既に開始時刻を過ぎた回、When 全体キャンセル、Then その回の status は変化しない。
- **AC-3 (冪等性)**: Given 既に全体キャンセル済みのシリーズ、When 再度全体キャンセル、Then 200 が返り状態は変わらない。
- **AC-4 (未存在)**: Given 存在しない `series_id`、When 全体キャンセル、Then 404。
- **AC-5 (解放の確認)**: Given 未来回をキャンセル後、When 同時間帯で空き検索や新規予約、Then その会議室・時間帯は予約可能になっている。

---

### US-R05: 個別回のキャンセル
**As a** 一般予約者（P-1）
**I want** シリーズのうち都合が悪い1回だけを取り消したい
**So that** シリーズ全体は維持しつつ特定回だけスキップできる

**トレース**: FR-3.1, FR-3.2, C-1

**受け入れ基準:**
- **AC-1 (既存 API で個別キャンセル)**: Given シリーズ内のある回の `reservation_id`、When 既存の `POST /reservations/{reservation_id}/cancel`、Then その回のみ cancelled になり 200、他の回は active のまま。
- **AC-2 (冪等性の踏襲)**: Given 既に cancelled の回、When 再キャンセル、Then 200 で cancelled のまま（既存仕様踏襲）。
- **AC-3 (未存在)**: Given 存在しない `reservation_id`、When キャンセル、Then 404。
- **AC-4 (時間帯の解放)**: Given 個別回をキャンセル後、When 同時間帯に予約、Then 再予約できる。

## Epic 3: 照会・表示

### US-R06: 予約一覧・詳細でのシリーズ情報表示
**As a** 会議室運用担当（P-2） / API 連携開発者（P-3）
**I want** 予約の一覧・詳細でその予約がどのシリーズに属するか分かるようにしたい
**So that** 定期予約と単発予約を区別し、シリーズ単位で状況を把握できる

**トレース**: FR-4.1, FR-4.2, C-1, C-4

**受け入れ基準:**
- **AC-1 (series_id の付与)**: Given シリーズの一部である予約、When `GET /reservations/{id}`、Then レスポンスに当該 `series_id` が含まれる。
- **AC-2 (単発は null)**: Given 単発予約、When 取得、Then `series_id` は `null`。
- **AC-3 (一覧での表示)**: Given 複数の予約、When `GET /reservations`、Then 各要素に `series_id` フィールドが含まれる。
- **AC-4 (既存契約の不変)**: Given 既存クライアント、When 既存の単発予約 API を利用、Then リクエスト契約は不変で、追加された `series_id` フィールドは既存動作を壊さない。

---

### US-R07（任意）: シリーズ単位の照会
**As a** 一般予約者（P-1） / 会議室運用担当（P-2）
**I want** あるシリーズのメタ情報と全回をまとめて取得したい
**So that** シリーズ全体の状況を一目で確認できる

**トレース**: FR-4.3（設計段階で要否確定）

**受け入れ基準:**
- **AC-1 (シリーズ取得)**: Given 存在する `series_id`、When `GET /reservations/recurring/{series_id}`、Then シリーズのメタ情報（繰り返しルール・予約者）と全回の予約リストが返る（200）。
- **AC-2 (未存在)**: Given 存在しない `series_id`、When 取得、Then 404。

> 注: このストーリーは設計段階でスコープ確定。MVP に含めるかは Application Design で判断。

## Epic 4: 互換性維持（横断・非機能）

### US-R08: 既存の単発予約 API・既存テストの不変性維持
**As a** API 連携開発者（P-3） / 会議室運用担当（P-2）
**I want** 定期予約機能の追加後も既存の単発予約 API と既存テストが一切壊れないことを保証したい
**So that** 既存クライアントと既存の品質保証を維持したまま新機能を導入できる

**トレース**: C-1, C-2, C-3, C-4, NFR-1, NFR-4

**受け入れ基準:**
- **AC-1 (既存テスト全パス)**: Given 既存の pytest スイート、When 機能追加後に実行、Then 既存テストはすべてパスし、既存テストのソースは改変されていない。
- **AC-2 (単発 API 契約不変)**: Given 既存の `POST /reservations` 等、When 呼び出し、Then リクエスト/レスポンス契約（`series_id` の追加を除く後方互換な拡張のみ）が維持される。
- **AC-3 (半開区間ロジック不変)**: Given 重複防止判定、When 単発・定期いずれの予約でも、Then 隣接OK・重なりNG の半開区間ロジックが同一に適用される。
- **AC-4 (HTTP ステータス方針)**: Given 新規 API、When 各種結果、Then README のステータス方針（201/200/400/404/409/422）に従う。
- **AC-5 (テストインポート規約)**: Given 新規テスト、When 追加、Then 既存と同じ `brown.tests.conftest` インポート規約に従い、収集・実行が既存と同様に成立する。
